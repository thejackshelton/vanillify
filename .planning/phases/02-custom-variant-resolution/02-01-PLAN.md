<objective>
Create the custom variant parsing and resolution module that translates user-provided `@custom-variant` CSS definitions into UnoCSS `VariantObject[]` entries.

Purpose: This is the core translation layer for Phase 2 -- it bridges Tailwind v4's `@custom-variant` syntax with UnoCSS's programmatic variant registration API. Without this, custom variants like `ui-checked:bg-blue-500` cannot be resolved.

Output: `src/variants/` module with parser, resolver, and types -- fully tested via TDD.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service | parseCustomVariantCSS | mitigate | Limit CSS input string to 10000 characters before regex execution; regex has no catastrophic backtracking (linear scan with `[^)]+` bounded by parentheses) |
| T-02-02 | Tampering | createVariantObject | mitigate | Validate variant names match `[\w-]+` pattern only -- reject names containing CSS special characters that could produce malformed selectors |
| T-02-03 | Tampering | resolveCustomVariants | accept | Selector templates are user-provided and used in CSS output context only (not executed as code). Malformed selectors produce invalid CSS, not security vulnerabilities. Low risk for a build-time library. |
</threat_model>
